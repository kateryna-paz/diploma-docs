# 3. Вимоги до системи

## 3.1 Функціональні вимоги

### 3.1.1 Збір даних

| ID   | Вимога                                                                                                    |
| ---- | --------------------------------------------------------------------------------------------------------- |
| F-01 | Система має зчитувати метрики сервісу у форматі Prometheus (HTTP).                                        |
| F-02 | Система має читати лог-файли у форматі JSON (access logs Nginx/Apache).                                   |
| F-03 | Система має підтримувати два режими роботи: аналіз архівних логів та аналіз нових даних у реальному часі. |

### 3.1.2 Попередня обробка даних

| ID   | Вимога                                                                                                     |
| ---- | ---------------------------------------------------------------------------------------------------------- |
| F-04 | Система має формувати вікна ознак на основі часових відрізків (тривалість вікна задається у конфігурації). |
| F-05 | Система має нормалізувати числові ознаки перед подачею до моделі.                                          |
| F-06 | Параметри нормалізації мають зберігатися разом з моделлю та завантажуватися при перезапуску.               |

### 3.1.3 Виявлення аномалій

| ID   | Вимога                                                                                          |
| ---- | ----------------------------------------------------------------------------------------------- |
| F-07 | Система має реалізовувати алгоритм Isolation Forest для виявлення аномалій у числових метриках. |
| F-08 | Система має реалізовувати LSTM-автоенкодер для врахування часових залежностей між запитами.     |
| F-09 | Результати обох детекторів мають об'єднуватися у єдиний показник аномальності.                  |
| F-10 | Поріг, вище якого спостереження вважається аномалією, має налаштовуватися в конфігурації.       |

### 3.1.4 Пояснення результатів

| ID   | Вимога                                                                                                                |
| ---- | --------------------------------------------------------------------------------------------------------------------- |
| F-11 | Для кожної виявленої аномалії система має формувати перелік ознак, що найбільше вплинули на рішення (на основі SHAP). |
| F-12 | Результати з поясненнями мають зберігатися та бути доступні через REST API.                                           |

### 3.1.5 Сповіщення та інтерфейс

| ID   | Вимога                                                                                  |
| ---- | --------------------------------------------------------------------------------------- |
| F-13 | Система має відправляти HTTP-сповіщення про виявлену аномалію на заданий webhook-адрес. |
| F-14 | Система має надавати REST API для перегляду виявлених аномалій та стану системи.        |

### 3.1.6 Навчання моделі

| ID   | Вимога                                                                               |
| ---- | ------------------------------------------------------------------------------------ |
| F-15 | Система має підтримувати навчання моделі на історичних даних (мінімум за 7 діб).     |
| F-16 | Збережена модель має завантажуватися при наступному запуску без повторного навчання. |

## 3.2 Нефункціональні вимоги

### 3.2.1 Продуктивність

| ID    | Вимога                                                                                                       |
| ----- | ------------------------------------------------------------------------------------------------------------ |
| NF-01 | Час від надходження події до формування результату — не більше 60 секунд у потоковому режимі.                |
| NF-02 | Система має коректно працювати при навантаженні до 1 000 подій на хвилину на звичайному ноутбуці розробника. |
| NF-03 | Навчання моделі на 30-денному датасеті не має займати більше 4 годин.                                        |

### 3.2.2 Надійність

| ID    | Вимога                                                                                             |
| ----- | -------------------------------------------------------------------------------------------------- |
| NF-04 | При відмові LSTM-компонента система має продовжувати роботу, використовуючи лише Isolation Forest. |
| NF-05 | Стан моделі та параметри обробки мають відновлюватися після перезапуску без втрати налаштувань.    |

### 3.2.3 Безпека

| ID    | Вимога                                           |
| ----- | ------------------------------------------------ |
| NF-06 | REST API має бути захищений токеном авторизації. |

### 3.2.4 Розгортання

| ID    | Вимога                                                          |
| ----- | --------------------------------------------------------------- |
| NF-07 | Система має запускатися через Docker Compose одною командою.    |
| NF-08 | Система має працювати на Linux та Windows з Python 3.10 і вище. |
